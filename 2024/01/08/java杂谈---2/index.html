String类

  • final修饰、实现了Comparable接口
  • String和包装类都具有不变性，对应得改变操作是新生成了一个对象，然后再指向它。将其作为参数进行函数传递时，函数类得改变并不会影响其值。相当于函数类得参数指向了另一个对象，而函数外得对象指向并没有变。单个对象是线程安全的。
  • ISO8859-1把所有的字符都当做一个byte处理，处理不了多个字节，故不能编码中文字符。GBK每一个中文都是对应2个字节；utf-8常规的中文都是3个字节。
  • 不乱码：（1）保证编码与解码的字符集名称一样（2）不缺字节。（3）字符集支持所有字符，，ISO8859-1本身就不支持汉字，因此会乱码。
  • 字符串拼接
    • 常量“+”常量，结果存储于常量池中。
    • 只要有变量，结果就是变量，存储于堆中。创建一个StringBuilder的实例，通过append()添加字符串，最后调用toString()返回一个字符串。（toString()内部new 一个String的实例）
    • 使用contact函数，无论是啥，结果都是变量，存储在堆中。
    • 拼接后调用intern()方法：返回值在常量池中
    • String:不可变的字符序列;底层使用char[] （jdk8及之前），底层使用byte[] （jdk9及之后）
    • StringBuffer:可变的字符序列;JDK1.0声明，线程安全的，效率低;底层使用char[] （jdk8及之前），底层使用byte[] （jdk9及之后）
    • StringBuilder:可变的字符序列;JDK5.0声明，线程不安全的,效率高;底层使用char[] （jdk8及之前），底层使用byte[] （jdk9及之后）
    • 这段内容描述了String类的当前实现方式：它使用一个char数组存储字符，对每个字符使用两个字节（十六位）。通过从许多不同应用程序收集的数据表明，字符串是堆内存使用的主要组成部分，并且大多数String对象仅包含Latin-1字符。这些字符仅需要一个字节的存储空间，因此这些String对象内部的char数组有一半的空间处于未使用状态。对于英文字符（ASCII字符）来说，每个字符都会占用两个字节的内存空间，这导致了在表示纯英文字符串时会浪费一半的内存空间。
    • 我们建议将String类的内部表示方式从UTF-16的char数组更改为一个字节数组加上一个编码标志字段。新的String类将根据字符串的内容，将字符存储为ISO-8859-1/Latin-1（每个字符一个字节）或UTF-16（每个字符两个字节）。编码标志将指示使用的编码方式。

集合框架

  • 数组存储多个数据方面的特点：

    数组一旦初始化，其长度就是确定的。
    数组中的多个元素是依次紧密排列的，有序的，可重复的
    (优点) 数组一旦初始化完成，其元素的类型就是确定的。不是此类型的元素，就不能添加到此数组中。

  • java.util.Collection:存储一个一个的数据。
    • 子接口：List:存储有序的、可重复的数据 (“动态”数组)
      • ArrayList:List的主要实现类；线程不安全的、效率高；底层使用Object[]数组存储。在添加数据、查找数据时，效率较高；在插入、删除数据时，效率较低。ArrayList 的扩容策略是增加当前容量的 50%。即*1.5
      • LinkedList:底层使用双向链表的方式进行存储；在对集合中的数据进行频繁的删除、插入操作时，建议使用此类在插入、删除数据时，效率较高；在添加数据、查找数据时，效率较低；
      • Vector:List的古老实现类；线程安全的、效率低；底层使用Object[]数组存储，Vector 的扩容策略是增加一倍。*2
      • 向List中添加元素的要求：要求元素所属的类一定要重写equals()!
    • 子接口：Set:存储无序的、不可重复的数据(高中学习的集合)
      • HashSet：主要实现类；底层使用的是HashMap，即使用数组+单向链表+红黑树结构进行存储。（jdk8中）
        • LinkedHashSet：是HashSet的子类；在现有的数组+单向链表+红黑树结构的基础上，又添加了组双向链表，用于记录添加元素的先后顺序。即： 我们可以按照添加元素的顺序实现遍历。便于频繁的查询操作。
      • TreeSet：底层使用红黑树存储。可以按照添加的元素的指定的属性的大小顺序进行遍历。
      • 无序性（Unordered）：无序性并不意味着随机性。在集合中，无序性指的是元素在集合中的存储或迭代顺序不是按照元素被添加的顺序或特定规则排列的。换句话说，元素的排列顺序不具有可预测性，不是依次紧密排列的，而是根据其哈希值计算的存储位置。这使得元素在集合中的存储顺序不可预测。
      • 不可重复性（Uniqueness）：在集合中，每个元素都是唯一的。对于 Set 类型的集合，其中的元素不能重复。判断元素是否相同通常依赖于元素的 hashCode() 方法和 equals() 方法。如果两个元素的 hashCode() 值相同且 equals() 方法返回 true，那么这两个元素被认为是相同的，因此 Set 集合中只会保留其中一个。
      • 添加到HashSet/LinkedHashSet中元素的要求:要求元素所在的类要重写两个方法：equals() 和 hashCode()。同时，要求equals() 和 hashCode()要保持一致性！我们只需要在IDEA中自动生成两个方法的重写即可，即能保证两个方法的一致性。
      • HashSet底层就是HashMap；LinkedHashSet底层就是LinkedHashMap；TreeSet底层就是TreeMap
  • java.util.Map:存储一对一对的数据(key-value键值对，(x1,y1)、(x2,y2) –> y=f(x),类似于高中的函数)，先判断hashcode，相同时再调用equals方法。后面的覆盖前面的。hashcode相同不一定是相同的对象。hashcode不同一定是不同的对象。
    • HashMap:主要实现类;线程不安全的，效率高;可**以添加null的key和value值;**底层使用数组+单向链表+红黑树结构存储（jdk8）
      • JDK7：HashMap内部声明了Entry，实现了Map中的Entry接口。（key，value作为Entry的两个属性出现）
      • JDK8：HashMap内部声明了Node，实现了Map中的Entry接口。（key，value作为Node的两个属性出现
      • key用于计算hash码得到存储的桶的位置，value存储在与key相关联的entry/node对象中。每个节点包含一个key-value对。
        • LinkedHashMap:是HashMap的子类；在HashMap使用的数据结构的基础上，增加了一对双向链表，用于记录添加的元素的先后顺序，进而我们在遍历元素时，就可以按照添加的顺序显示。开发中，对于频繁的遍历操作，建议使用此类
    • TreeMap:底层使用红黑树存储;可以按照添加的key-value中的key元素的指定的属性的大小顺序进行遍历。需要考虑使用①自然排序 ②定制排序。
    • Hashtable:古老实现类;线程安全的，效率低;不可以添加null的key或value值;底层使用数组+单向链表结构存储（jdk8）
      • Properties:其key和value都是String类型。常用来处理属性文件。
  • 集合的父类Collection接口，注意Map是和它同级别的接口。

ArrayList与Vector

  • ArrayList是新版的动态数组，线程不安全，效率高，Vector是旧版的动态数组，线程安全，效率低。
  • 动态数组的扩容机制不同，ArrayList默认扩容为原来的1.5倍，Vector默认扩容增加为原来的2倍。
  • 数组的初始化容量，如果在构建ArrayList与Vector的集合对象时，没有显式指定初始化容量，那么Vector的内部数组的初始容量默认为10，而ArrayList在JDK 7.0 及之前的版本也是10，JDK8.0 之后的版本ArrayList初始化为长度为0的空数组，之后在添加第一个元素时，再创建长度为10的数组。饿汉式和懒汉式

ArrayList与数组

  1. 固定大小 vs. 动态大小：
    • 数组： 数组在创建时需要指定大小，且大小固定。一旦创建后，其大小不能动态改变。
    • ArrayList： ArrayList 是基于动态数组实现的集合类，它在创建时可以为空，随着元素的添加，其大小会动态增长。它会自动处理容量的调整。
  2. 类型：
    • 数组： 数组可以存储任意类型的元素，包括基本数据类型和对象。
    • ArrayList： ArrayList 只能存储对象（引用类型），而不支持基本数据类型。Java提供了对应的包装类来处理基本数据类型，比如 Integer 代表 int。
  3. 长度变化：
    • 数组： 数组长度一旦确定，不能直接改变。如果需要改变长度，需要创建一个新的数组，并将元素拷贝到新数组中。
    • ArrayList： ArrayList 可以根据需要动态增加或缩减容量，内部实现会自动处理。这使得添加和删除元素更加灵活。
  4. 性能：
    • 数组： 数组在访问元素时具有非常快的速度，因为它们是连续的内存块。但在插入和删除元素时，可能需要移动大量元素。
    • ArrayList： 在访问元素时，ArrayList 的性能与数组类似。但在添加和删除元素时，由于其动态调整大小的能力，性能可能更好，尤其是在末尾添加或删除元素时。
  5. API 和灵活性：
    • 数组： 数组的操作相对简单，没有提供像集合类那样的丰富方法。长度固定，不支持高级方法。
    • ArrayList： ArrayList 提供了丰富的方法和API来操作集合，包括添加、删除、查找、排序等，使其更加灵活和易用。

HashMap中数据添加过程

计算hashcode–>扰动函数（hash）–>计算index–>比较hash值–>相同则再比较equals–> 插入或者覆盖–>转为红黑树或者扩容操作。 先准备插入之后才进行扩容或转化过程，奥里给。

hash过程

hash是具有唯一性且不可逆的。

将(key1,value1)添加到当前hashmap的对象中。首先会调用key1所在类的hashCode()方法，计算key1的哈希值1，
此哈希值1再经过某种运算(hash())，得到哈希值2。此哈希值2再经过某种运算(indexFor())，确定在底层table数组中的索引位置i。

table数组长度

一定是2的n次幂。因为如果数组的长度为2的n次幂，那么table.length-1的二进制就是一个高位全是0，低位全是1的数字，这样才能保证每一个下标位置都有机会被用到。算索引是&运算。

index冲突

虽然从设计hashCode()到上面HashMap的hash()函数，都尽量减少冲突，但是仍然存在两个不同的对象返回的hashCode值相同，或者hashCode值就算不同，通过hash()函数计算后，得到的index也会存在大量的相同，因此key分布完全均匀的情况是不存在的。那么发生碰撞冲突时怎么办？

红黑树

红黑树是一种自平衡的二叉搜索树（BST），属于平衡二叉查找树的一种。

  1. 节点颜色： 节点要么是红色，要么是黑色。
  2. 根节点和叶节点（空节点）： 根节点是黑色的，叶节点（是不存储数据得空节点）也是黑色的。
  3. 红色节点规则： 不能有两个相邻的红色节点，即红色节点不能相连。即红色节点得父亲和孩子都是黑色得。
  4. 路径规则： 从任一节点到其每个叶子的所有路径都包含相同数量的黑色节点（即保持黑色平衡）。
  5. 树的高度趋近于log（n）。树得高度最高为2log（n+1），故对其进行增删查操作得时间复杂度为log（n）
  6. 插入
    • 插入新节点： 将新节点插入到红黑树中，通常将新节点插入为红色节点。
    • 修正红黑树性质：
      • 情况1： 如果父节点是黑色，则不需要额外操作，插入一个红色节点不会违反红黑树性质。
      • 情况2：如果父节点是红色，那么需要进一步考虑：
        • 如果叔节点（父节点的兄弟节点）也是红色，此时需要做的是将父节点和叔节点的颜色改变，并检查祖父节点是否违反了红黑树的性质。这时需要将祖父节点视作新插入的节点，再进行修正。
        • 如果叔节点是黑色或者空节点（NIL），需要进行旋转和重新着色操作，以保持红黑树的平衡性。
          • 左旋或右旋：通过旋转重新调整节点，以保持树的平衡。
          • 重新着色：对节点及其父节点、祖父节点进行颜色变换，使得红黑树的性质得以维持。
    • 根节点的颜色处理： 最后，确保根节点是黑色。如果在修正性质的过程中改变了根节点的颜色，需要将根节点的颜色设为黑色，以符合红黑树的性质。
  7. 删除
    • 找到要删除的节点： 首先，找到要删除的节点。
    • 处理删除节点的子节点情况：
      • 情况1： 如果待删除节点是叶子节点或只有一个子节点，可以直接删除。
      • 情况2： 如果待删除节点有两个子节点，需要找到其后继节点（即右子树中最小的节点，或者左子树中最大的节点）作为替代，并将后继节点的值复制到待删除节点，然后删除后继节点。
    • 调整红黑树以保持平衡：
      • 情况1： 如果删除的节点是红色的，不会破坏红黑树的性质，不需要额外的平衡操作。
      • 情况2：如果删除的节点是黑色的，可能破坏了红黑树的性质，需要进行调整以保持平衡。
        • 情况2a： 被删除节点为黑色，其子节点为红色。此时，将子节点染成黑色即可满足性质。
        • 情况2b： 被删除节点和其子节点都为黑色。这种情况下需要进行调整，可能涉及旋转和颜色变换，以确保树的性质不被破坏。
    • 根节点的颜色处理： 确保删除节点后，根节点依然是黑色，以满足红黑树的性质。

扩容

key不可变性

key-value存储到HashMap中会存储key的hash，是将hashcode进行干扰后的hash值，这样就不用在每次查找时重新计算每一个Entry或Node（TreeNode）的hash值了，因此如果已经put到Map中的key-value，再修改key的属性，而这个属性又参与hashcode值的计算，那么会导致匹配不上。

当你将作为key的对象的属性修改后，原本存储在HashMap中的位置会因为哈希码变化而改变，导致在之后无法准确找到该对象对应的值。这可能导致在HashMap中无法正确检索到该key对应的value，即使它实际上仍在HashMap中。

HashMap循环链表 jdk7

避免HashMap发生死循环的常用解决方案：

  • 多线程环境下，使用线程安全的ConcurrentHashMap替代HashMap，推荐
  • 多线程环境下，使用synchronized或Lock加锁，但会影响性能，不推荐
  • 多线程环境下，使用线程安全的Hashtable替代，性能低，不推荐

HashMap死循环只会发生在JDK1.7版本中，主要原因：头插法+链表+多线程并发+扩容。

在JDK1.8中，HashMap改用尾插法，解决了链表死循环的问题。

hashcode与equals

equals相同，则hashcode相同；hashcode相同，不一定equals相同。

  1. hashCode() 方法：
    • hashCode() 方法用于获取对象的哈希码值。
    • 哈希码是根据对象的内部状态计算出的一个整数，在集合类如哈希表（HashMap、HashSet 等）中用于快速定位对象。
    • 这个值通常用于确定对象在哈希表中的存储位置，以便快速访问对象。
    • hashCode() 方法是在 Object 类中定义的，子类可以选择重写以提供更准确和有效的哈希码计算。
  2. equals() 方法：
    • equals() 方法用于比较两个对象是否在逻辑上相等。
    • 默认情况下，equals() 方法比较的是对象的引用，即检查两个对象是否指向内存中的同一位置（与 == 操作符类似）。
    • 通常需要在自定义类中重写 equals() 方法，根据对象的内部状态来判断相等性。
    • 重写 equals() 方法时，需要满足一定的约定，例如自反性、对称性、传递性、一致性和非空性。
      功能和用途不同：
  • hashCode()：返回对象的哈希码值，用于快速确定对象在哈希表等数据结构中的存储位置。
  • equals()：用于比较两个对象是否在逻辑上相等，通常需要根据对象的内部状态来重写这个方法。

返回值和类型不同：

  • hashCode()：返回一个 int 类型的哈希码值。
  • equals()：返回一个 boolean 类型的结果，表示两个对象是否相等。

原始实现和需求不同：

  • hashCode()：在 Object 类中有默认的实现，但通常需要在自定义类中重写以提供更准确和有效的哈希码计算。
  • equals()：在 Object 类中使用了对象的引用比较，但通常需要在自定义类中根据业务逻辑重写。

基础数据结构关联不同：

  • hashCode()：与哈希表等数据结构直接关联，用于快速定位存储位置。
  • equals()：通常用于集合类中的去重和查找操作，决定两个对象在集合中是否被视为相同。

重写需求和影响不同：

  • hashCode()：若自定义类需要作为键存储在哈希表等结构中，通常需要重写以确保相等对象具有相等的哈希码值。
  • equals()：对于自定义类，如果需要根据对象的内部状态来定义相等性，就需要重写这个方法。

TCP

TCP是一种面向连接的、可靠的、基于字节流的传输层协议，用于在网络中可靠地传输数据。它提供了数据传输的顺序性、可靠性和流量控制等特性，适用于许多网络通信场景。

三次握手

TCP协议中，在发送数据的准备阶段，客户端与服务器之间的三次交互，以保证连接的可靠。

  • 第一次握手，客户端向服务器端发起TCP连接的请求
  • 第二次握手，服务器端发送针对客户端TCP连接请求的确认
  • 第三次握手，客户端发送确认的确认

两次握手

  1. 可能存在旧连接问题：
    • 在两次握手中，客户端发送连接请求（SYN），服务器收到后回复连接确认（ACK+SYN）。但如果这个ACK+SYN的响应在网络中滞留，导致客户端没有收到确认，客户端会重新发送连接请求（SYN）。如果这个延迟的ACK+SYN最终到达客户端，但此时服务器已经接受了第二次发送的SYN并建立了连接，就会出现旧连接，可能导致数据混乱和安全问题。
  2. 无法验证客户端能否接收响应：
    • 两次握手不包含客户端对服务器的确认，即服务器发送的ACK（确认）并不包含客户端对该确认的响应。这意味着，服务器无法确认客户端是否真正收到了连接确认。这种情况下，可能会有一些网络问题导致连接无法建立。

四次挥手

  • 第一次挥手：客户端向服务器端提出结束连接，让服务器做最后的准备工作。此时，客户端处于半关闭状态，即表示不再向服务器发送数据了，但是还可以接受数据。
  • 第二次挥手：服务器接收到客户端释放连接的请求后，会将最后的数据发给客户端。并告知上层的应用进程不再接收数据。
  • 第三次挥手：服务器发送完数据后，会给客户端发送一个释放连接的报文。那么客户端接收后就知道可以正式释放连接了。
  • 第四次挥手：客户端接收到服务器最后的释放连接报文后，要回复一个彻底断开的报文。这样服务器收到后才会彻底释放连接。这里客户端，发送完最后的报文后，会等待2MSL，因为有可能服务器没有收到最后的报文，那么服务器迟迟没收到，就会再次给客户端发送释放连接的报文，此时客户端在等待时间范围内接收到，会重新发送最后的报文，并重新计时。如果等待2MSL后，没有收到，那么彻底断开。

三次挥手

  1. 数据丢失或半关闭连接：
    • 在三次挥手中，客户端发送了关闭连接请求（FIN），服务器确认后也发送了关闭连接的确认（ACK+FIN），此时连接处于半关闭状态。
    • 如果此时客户端接收到服务器的FIN并立即关闭了连接，可能导致服务器尚未发送的数据丢失。
  2. 数据包的延迟或混乱：
    • 如果在三次挥手过程中，有延迟的数据包未被处理或传输，可能会导致连接未能完整地关闭。这可能会产生混乱的状态，使得某些数据丢失或无法按序到达。
  3. 无法确保完整性：
    • 三次挥手无法完全确保对方已经接收到了最后的确认信息，可能存在一定的不确定性。

TCP与UDP的区别

连接性：

  • TCP：
    • 面向连接：建立连接、传输数据、关闭连接。
    • 通过三次握手建立连接，四次挥手关闭连接。
  • UDP：无连接：不需要建立连接，直接传输数据。

可靠性：

  • TCP：
    • 可靠性高：提供数据确认、重传机制，保证数据的完整性和顺序性。
    • 数据不会丢失或损坏，且按序到达。
  • UDP：
    • 不可靠性高：没有确认和重传机制，数据传输不保证可靠性。
    • 数据可能丢失或乱序到达。

流量控制：

  • TCP：有流量控制机制，通过调节发送数据的速率来控制传输流量，避免网络拥塞。
  • UDP：没有流量控制，数据被发送后不会考虑网络状态，可能导致网络拥塞。

适用场景：

  • TCP：适用于对数据完整性和顺序性要求高的场景，如文件传输、网页访问、电子邮件发送等。
  • UDP：适用于实时性要求高、对数据完整性要求相对较低的场景，如视频流、音频通话、在线游戏等。

效率：

  • TCP：传输完毕，需释放已建立的连接，效率低
  • UDP：发送数据结束时无需释放资源，开销小，通信效率高

域名/ip

    • 顶级域（Top-Level Domain，TLD）：比如.com、.org、.net等。它是域名层次结构中的最高层，用于表示域名的类别或国家/地区代码。
    • 二级域名（Second-Level Domain）：位于顶级域名之前的部分，通常用于标识特定的组织、公司或个人。
    • 子域名（Subdomain）：位于二级域名之前的部分，可以进一步细分和标识特定的部门、服务或位置等。
  • 域名在其特定的顶级域下应该是唯一的，这样可以确保互联网上的资源可以通过独一无二的标识符被正确地识别和访问。
  • 一个ip可以对应多个域名
  • 一个域名通常对应1个ip。然而，有时一个域名可能对应多个IP地址。这种情况可以是为了负载均衡或高可用性，其中一个域名被映射到多个服务器的不同IP地址上，以提供更好的性能和容错能力。这样可以分担流量或在一个服务器不可用时转向另一个服务器。
  • IP地址可以唯一标识网络中的设备，那么端口号就可以唯一标识设备中的进程（应用程序）。
  • 端口号：用两个字节表示的整数，它的取值范围是0~65535。
    • 公认端口：0~1023。被预先定义的服务通信占用，如：HTTP（80），FTP（21），Telnet（23）
    • 注册端口：1024~49151。分配给用户进程或应用程序。如：Tomcat（8080），MySQL（3306），Oracle（1521）。
    • 动态/ 私有端口：49152~65535。

反射

反射是一种在运行时检查或操作类、方法、属性等程序结构的能力。在Java中，反射允许你在程序运行时检查类的信息、获取类的构造函数、方法和字段，以及在运行时调用方法和修改字段的值。

  1. 获取类的信息：可以动态地获取类的名称、方法、构造函数和字段等信息。
  2. 动态创建对象：在运行时根据类名或类型信息创建类的对象。
  3. 动态调用方法：可以在运行时调用类的方法，包括公共、私有和静态方法。
  4. 访问和修改字段：可以获取和修改类的字段值，包括公共、私有和静态字段。

主要api

  • java.lang.Class：代表一个类
    java.lang.reflect.Method：代表类的方法
    java.lang.reflect.Field：代表类的成员变量
    java.lang.reflect.Constructor：代表类的构造器
    … …

优点：

  • 提高了Java程序的灵活性和扩展性，降低了耦合性，提高自适应能力
  • 允许程序创建和控制任何类的对象，无需提前硬编码目标类

缺点：

  • 反射的性能较低。
    • 反射机制主要应用在对灵活性和扩展性要求很高的系统框架上
  • 反射会模糊程序内部逻辑，可读性较差。

实现

反射是通过 Java 的类加载机制和 java.lang.reflect 包中的类来实现的。Java 的类加载机制负责加载类的字节码并创建对应的 Class 对象，而 java.lang.reflect 包中的类提供了一系列方法和工具，允许程序在运行时检查和操作类、方法、字段等。

在实现反射时，以下是主要的步骤和机制：

  1. 类加载：首先，Java 类加载器负责加载类的字节码文件，并将其转换为 Class 对象。类加载器可以是系统默认的类加载器，也可以是自定义的类加载器。
  2. 获取 Class 对象：通过 Class.forName("ClassName")、getClass() 方法或类字面常量等方式获取类的 Class 对象。
  3. 获取类的信息：通过 Class 对象提供的方法，例如 getMethods()、getDeclaredMethods()、getFields()、getConstructors() 等，可以获取类的方法、字段、构造方法等信息。
  4. 调用方法和操作字段：通过 Method、Field、Constructor 等反射类提供的方法，可以动态调用类的方法、获取和设置字段的值，以及创建类的实例。
  5. 动态创建对象：利用 Constructor 类的 newInstance() 方法可以在运行时动态地创建类的实例。

创建对象

使用反射创建对象的实例会调用构造函数，clone和反序列化的方式不会调用构造函数。

Objenesis是一个开源的Java库，用于实例化对象，而不必调用构造函数。它提供了一种绕过构造函数直接实例化对象的机制。通常情况下，通过Java的new关键字或者通过反射调用构造函数来实例化对象都会触发构造函数的执行，但是Objenesis可以通过其他方式在不调用构造函数的情况下创建对象。Objenesis主要使用了一些Java特性（如序列化、反序列化、字节码操作等）来绕过构造函数创建对象。它适用于某些特定场景，比如在某些测试框架、序列化库或者依赖注入容器中。通过Objenesis，可以创建对象并绕过构造函数的执行，这在某些情况下可能是有用的。

通过new创建对象的效率高于使用反射的方式：使用 new 关键字直接创建对象时，编译器在编译时可以直接识别到要实例化的类，因此在运行时只需根据类的结构调用构造函数，这个过程是直接的、高效的。通过反射创建对象会涉及到更多的步骤和操作，需要动态地查找类、获取构造函数、调用构造函数等过程，这些步骤都需要在运行时进行。因此，反射创建对象相比直接使用 new 关键字会更耗费时间，并且性能相对较低。

Lambda表达式

Lambda表达式主要用于函数式接口，函数式接口是只有一个抽象方法的接口，当然该接口可以包含其他非抽象方法。。Lambda 表达式：在Java 8 语言中引入的一种新的语法元素和操作符。这个操作符为 “->” ， 该操作符被称为 Lambda 操作符或箭头操作符。它将 Lambda 分为两个部分：

  • 左侧：指定了 Lambda 表达式需要的参数列表
  • 右侧：指定了 Lambda 体，是抽象方法的实现逻辑，也即 Lambda 表达式要执行的功能。

语法格式一：无参，无返回值
语法格式二：Lambda 需要一个参数，但是没有返回值。
语法格式三：数据类型可以省略，因为可由编译器推断得出，称为“类型推断”
语法格式四：Lambda 若只需要一个参数时，参数的小括号可以省略
语法格式五：Lambda 需要两个或以上的参数，多条执行语句，并且可以有返回值
语法格式六：当 Lambda 体只有一条语句时，return 与大括号若有，都可以省略

  • 在语法格式三 Lambda 表达式中的参数类型都是由编译器推断得出的。Lambda 表达式中无需指定类型，程序依然可以编译，这是因为 javac 根据程序的上下文，在后台推断出了参数的类型。Lambda 表达式的类型依赖于上下文环境，是由编译器推断出来的。这就是所谓的“类型推断”。

函数式接口

  • 只包含一个抽象方法（Single Abstract Method，简称SAM）的接口，称为函数式接口。当然该接口可以包含其他非抽象方法。
  • 你可以通过 Lambda 表达式来创建该接口的对象。（若 Lambda 表达式抛出一个受检异常(即：非运行时异常)，那么该异常需要在目标接口的抽象方法上进行声明）。
  • 我们可以在一个接口上使用 @FunctionalInterface 注解，这样做可以检查它是否是一个函数式接口。同时 javadoc 也会包含一条声明，说明这个接口是一个函数式接口。
  • 在java.util.function包下定义了Java 8 的丰富的函数式接口

方法引用

当要传递给Lambda体的操作，已经有实现的方法了，可以使用方法引用！
方法引用可以看做是Lambda表达式深层次的表达。换句话说，方法引用就是Lambda表达式，也就是函数式接口的一个实例，通过方法的名字来指向一个方法，可以认为是Lambda表达式的一个语法糖。
语法糖（Syntactic sugar），也译为糖衣语法，是由英国计算机科学家彼得·约翰·兰达（Peter J. Landin）发明的一个术语，指计算机语言中添加的某种语法，这种语法对语言的功能并没有影响，但是更方便程序员使用。通常来说使用语法糖能够增加程序的可读性，从而减少程序代码出错的机会。

格式

格式：使用方法引用操作符 “::” 将类(或对象) 与 方法名分隔开来。
如下三种主要使用情况：

  • 情况1：对象 :: 实例方法名
  • 情况2：类 :: 静态方法名
  • 情况3：类 :: 实例方法名

使用前提

要求1：Lambda体只有一句语句，并且是通过调用一个对象的/类现有的方法来完成的
要求2：
针对情况1：函数式接口中的抽象方法a在被重写时使用了某一个对象的方法b。如果方法a的形参列表、返回值类型与方法b的形参列表、返回值类型都相同，则我们可以使用方法b实现对方法a的重写、替换。对象 :: 实例方法名

针对情况2：函数式接口中的抽象方法a在被重写时使用了某一个类的静态方法b。如果方法a的形参列表、返回值类型与方法b的形参列表、返回值类型都相同，则我们可以使用方法b实现对方法a的重写、替换。类 :: 静态方法名

针对情况3：函数式接口中的抽象方法a在被重写时使用了某一个对象的方法b。如果方法a的返回值类型与方法b的返回值类型相同，同时方法a的形参列表中有n个参数，方法b的形参列表有n-1个参数，且方法a的第1个参数作为方法b的调用者，且方法a的后n-1参数与方法b的n-1参数匹配（类型相同或满足多态场景也可以）类 :: 实例方法名

构造函数引用

当Lambda表达式是创建一个对象，并且满足Lambda表达式形参，正好是给创建这个对象的构造器的实参列表，就可以使用构造器引用。

格式：类名::new

数组构造引用

当Lambda表达式是创建一个数组对象，并且满足Lambda表达式形参，正好是给创建这个数组对象的长度，就可以数组构造引用。
格式：数组类型名::new
